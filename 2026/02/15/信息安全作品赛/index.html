信息安全作品赛 赛前准备

想尝试信息安全作品赛，在稍微了解了一下作品赛之后发现和信息安全攻防赛差距挺大的，并不是指CTF和作品赛的区别，而是学习范围和比赛逻辑都有许多的不同。

作品赛一直的问题就是操作空间大到离谱，而且评判的主观性过强，选题大于一切等请况，让这个比赛直接成为了PPT大赛。我感觉主要原因就是光凭大学生的能力以及目的只是为了拿奖，其实就不会做出特别好用的作品。

因为你写的每一行代码评委基本上都不会看，做过代码审计就知道等你审计完了，作品就是你的了。于是重点从技术变为了选题，做好PPT然后在评委面前吹NB，然后回答评委的问题。

最近两届的作品赛我去网上搜索作品都是查不到的，也很难有一些参考。然后我又想了一下作品赛无非就是一个加强版的网络安全毕设。如果按这个方向开发似乎没问题。

1.对于信息安全的理解

信息安全我可以拆分为信息处理和网络安全，因为一些和安全基本上不沾边的作品都有一等奖，当然是我太菜了看不懂选题而已

这个和我说的安全是不是有一点没关系，所以我只能认定这是信息处理类作品。所以说以我的认知来说我是看不懂一些获奖作品的选题的，应该是我对信息安全理解的不够透彻。

我去查找了选题类似的文章有一篇硕士的毕业论文，其中的大概意思就是通过爬虫获取网络数据然后通过情感词典来对数据进行分析检测，以达到阻止网络暴力的目的，虽然选题是近几年比较，但是我看试看论文的时候感觉这就是将计算机的爬虫和分析用到了文科的论文里面。说白了爬虫上限很高下限很低而且主流网站的爬虫网上一找一大堆，分析也挺抽象的总之这技术基本上是网上简单开源的。

也可能是这种请况，之前班级群里面要求去做一个网络安全的答题，我想终于到我展示的时候了，结果看了题目才发现就是一个防诈骗的哪种智障答题。我称这种为广义的网络安全。于是我又联想到了为什么这些选题要剑走偏锋，我只能说可能不是计算机专业或者以前没有进行过专门准备临时准备参加这个比赛的（虽然我也是临时准参加）。因为这些主要感觉就是文科进行数据分析处理的那一方面

总的来说这个比赛感觉不是很难，我认为如果这些作品都是有很强的实力的话，我和我的学校都吃屎去吧，打CTF的都小丑了真正的高手在作品赛。

在早几年前的时候区块链和各种加密算法十分流行，但是近两年基本上看不到了，只能说明选题被选烂了，觉得不能选择太大众的选题。然后我仔细观察了一下一些稳一点的选题比如

这只是一部分，仔细观察就可以知道只要偏向技术方面的只要不是那些被选烂了的古早选题，基本上就不要去擦边。

最后看下来，我对那些各种信息分析的作品有一点生理上的排斥，感觉有点为了答辩和拿奖去硬要擦信息安全的边，我也不知道什么是评委口中的创新和答辩时面对那些擦边选题会问一些什么问题。

我又想了一个问题，这些有意思的选题到底是在什么请况下决定的，既然我知道要去选择一些稳一点的选题，为什么其他作品就敢选一些看不懂的题目。这就是学校的力量吗。

然后对比了十七届和十八届的获奖作品名单，发现一个问题，上面那种我认为拿一些简单的数据分析来碰瓷的变少了，作品的技术性更强了。然后就是感觉网络安全方面的作品变多了，主要是溯源和漏洞挖掘这些方面的。然后网络安全和信息数据分析的这两大类的共同特征就是十八届相较于十七届基于大模型的作品数量基本上实现了超级加倍，从一等奖到优胜奖都是这样的。

接入大模型这可能就是25年所谓的创新点，LLM，AI-Agent，AIGC检测等这些ai技术。那我们26年的创新点是什么，我感觉还是ai虽然ai被作为选题有几年了，但是除了ai感觉没其他的了这两年。

2.选题分析

除了获奖的作品，那些优胜将也需要看一下。我认为如果可以通过线上初赛的话作品完成度应该是有保障的（有保障但是不多肯定有因为没做完就被刷的），初赛没过的应该就是完成度的问题。当然决赛上相同类型选题的作品肯定还是会受到完成度的影响，但是感觉还是选题占的原因大一点。

我对自己看不懂的选题就不管了，十八届还是出现了区块链和国密算法这些老选题。

然后就是一些简单的漏扫系统，那种作品无非就是一个普通网络安全专业的毕业设计，无论是从完成度还是使用上面都是没问题，但是没有使用到什么技术手段，也没有什么创新点。我感觉就是作品赛参赛指南之好看的前端界面和PPT，太明显肯定是会暴毙的。

也就是说我们不能把这个比赛认为的太水，毕竟自己都感觉肯定不行估计是不行的。还是需要认真对待。

我们选题感觉需要注意几点

  1. 我们三个全是web手，只能去做漏洞扫描和溯源方面的内容。
  2. 别人都基于大模型我们也必须有，技术和实力比不过别人就需要防止被直接毙掉。
  3. 使用最先进的大模型技术，ai强大总没坏处。
  4. 尽量根据实际请况决定，比如漏扫把常见漏洞都扫会导致作品完成度的问题，就减少扫描种类甚至只扫一种也是可以的

我感觉这个就是我们的一个最最最理想但是也很难达成的请况。

然后从25年到26年，ai的发展完全超出了我的预计，至少24，25这两年感觉属于ai发展的元年。模型的升级估计也是大大降低了后端的难度。

3.开发流程

先看下临时选题Spyware-基于大模型的Web漏洞扫描系统自动识别网站潜在漏洞并生成攻击验证与安全报告

选题选好了但是还是要具体一下需要实现的功能和开发流程以及如何实现一些用于获奖的优势

虽然现在的很多ai都可以自动渗透，但是ai的不确定性太多了，渗透功能还是我们自己使用代码实现。还是将ai当作那种问答式ai去使用。

其中项目基本流程大致为

其中大概就是我们把爬虫拿到的请求和相应转化为json格式然后利用提取信息和ai进行对话。

我们不需要特地的去配置和训练ai，大概就是我们需要提供给ai一些我们收集到的对于特定请况下使用的payload让ai根据我们提供的知识库自己去生成对应的payload。然后进行测试。

然后估计也是利用ai去自动生成漏洞报告。这方面除了传统的漏洞报告还可以生成一些比如攻击利用链以及防御建议等加分项后面再决定吧。

然后就是先决定好一个简易的开发流程

各个阶段解释一下就是

  • 阶段一，我们需要提前确定好我们整个项目的整体需求，需求需要满足可以根据需求开发出一个实现了核心功能的项目，不至于后期难以维护修改的请况，后续有时间加功能也方便一点，总结就是第一次就需要确定好整个项目的核心，后面就不要动核心了。
  • 阶段二，系统架构的设计，首先需要确定好技术栈，然后就是设计好项目结构主要是各个模块的功能要利于分工和独立开发，以及确定好整个项目的运行流程。项目结构肯定需要参考一些样本，尽量不要相信自己
  • 阶段三，开发核心功能，比如在接入ai之前需要开发好爬虫模块，渗透测试模块，数据导出模块等等，在没有接入ai之前都可以使用输出语句测试功能。
  • 阶段四，在可以实现项目核心功能的时候可以接入ai进行测试，完善功能。ai还没有具体学这个后面看看

后面几天的任务是去学习一下怎么进行ai编程以及阅读各个开源的类似大模型项目，看看高手是怎么设计项目的。开源项目比如这个GitHub - lintsinghua/DeepAudit: DeepAudit：人人拥有的 AI 黑客战队，让漏洞挖掘触手可及。国内首个开源的代码漏洞挖掘多智能体系统。小白一键部署运行，自主协作审计 + 自动化沙箱 PoC 验证。支持 Ollama 私有部署 ，一键生成报告。支持中转站。让安全不再昂贵，让审计不再复杂。 · GitHub可以参考一下

4.各模块开发

各个模块必须相互独立，不然就不好测试各个模块是否符合条件

1.crawler模块

这个是最简单的模块，只需要对一个url进行爬虫即可，我这里没有选择深度爬虫，只是把当前路由的信息给爬取了

这是buu上一道sql注入爬取数据示例

可以看到通过url只爬取到一个界面，实际上这个题目也只有一个界面。然后先是会汇总存在注入点的参数，然后各个page中存在注入的位置，同时也爬取了请求和响应中可以利用的数据。

2.engine模块

攻击模块的作用主要是为了让ai生成的payloads被我们的代码伪造请求发送出去。先通过爬虫获取网站信息以及参数，然后对这些参数进行payloads测试。

在没接入ai的时候是通过字典去进行测试的。

每个请求测试完之后需要将请求和响应发送到detector模块进行漏洞命中和总结漏洞信息。

然后我们可以在漏扫的时候不是把没一种漏洞都跑一遍，而是选择测试特定的漏洞调用特定的engine去进行测试

然后就是漏洞的风险评估，这是sql注入漏洞的一个标准，先放在这里

等级 触发条件 示例
HIGH 明确的 SQL 错误信息 “you have an error in your sql”, “sql syntax near”, “mysql server version”, HTTP 500, 状态码变化
MEDIUM 可疑的数据库相关关键字 “sql syntax”, “mysql_fetch”, “warning: mysql”, 响应时间 > 5s, 响应长度变化 > 30%
LOW 弱特征匹配 “mysql”, “sqlite”, “postgresql”, “mssql” 等通用数据库名称

3.ReAct模式进行渗透

随着我学习的深入，不可避免的以前的一些想法会被推翻。

之前就在纠结一个问题，如果是先自动去跑一遍字典然后去漏洞命中在渗透方面不就没有ai什么事情了吗。然后目前的问题有：

  • ai自己使用沙箱写脚本去进行渗透用来写题目还是可以，但是用在项目中及其不稳定。
  • payload不可能就是跑字典，需要有ai辅助进行渗透测试
  • ai自己从零开始生成payload成本过大，没必要，效果也不稳定

思考了一下和看了一下其他项目，我想到一种目前可行的方案，就是减少一开始跑字典测试的payloads数量，将跑出来的结果和响应给到ai，让ai在此基础上再生成一套升级绕过的payload。然后利用一个ReAct模式去重复这个payload升级的过程，重复3到5遍如果还没有结果就断定没有漏洞。

这样我们既保留了半自动渗透测试的稳定性，也让我们的渗透测试有了ai的参与。渗透能力不再取决于字典的大小，有了ai赋能不会因为环境等某些问题直接让字典中所有payload失效。然后使用了ReAct模式也会成为这个ai项目的亮点。

然后还有一些问题需要解决：

  • 扫描结果不可能全部交给ai处理，需要精确匹配可以利用的注入点和原始payload
  • 不能直接把整个请求和响应丢给ai，而是需要精确匹配数据
  • 每次不要都把ReAct跑完，可以提前命中就提前命中
  • 不要直接相信ai的payload，最后测试的时候需要验证payload是否可行才能命中漏洞

目前还没有具体实现，万一明天又觉得这个不行了也是有可能的。

5.Multi Agent

这个是该项目的第二个亮点，我想我如果只是拿ai去跑一下paylaod那就太简单了。

Multi Agent实际上也是分模块的原理，通过调度不同的Agent可以实现处理多个不同的任务，比起单个Agent更加灵活和可维护，功能也更加强大。

然后就是多智能体系统，虽然是多智能体，但是某些模块如果可以使用代码实现也是不需要使用到ai的，比如我们可以将信息探测的ai替换为爬虫代码然后作为工具去供ai使用，这样比起让ai去探测信息更加稳定和方便。

我们已经决定好要在payload的构建上使用ReAct模式，于是现在该决定整体的框架了。也就是这个项目作为一个Mulit-Agent系统的整个流程需要确定下来。

首先需要确定有哪些需要使用到的Agent。

0.Controller

Controller作为整个项目的控制器，控制着整个流程，我看其他项目中这个都是交给ai去实现的，我没这么厉害，还是按照原计划使用代码实现

职责：

  • 管理扫描流程
  • 决定是否进入AI
  • 控制ReAct循环次数
  • 管理上下文（context）

1.Recon Agent

这个Agent的作用就是去探测出注入点等信息，也就是要拿到渗透测试的入口，这个已经交给了爬虫去干了，也没有必要使用ai。

2.Probe Agent

测试Agent，我的项目不是提供给ai沙箱要它自己去写脚本跑，而是ai提供payload然后通过代码去进行渗透测试，属于半自动的。

然后最开始的请求不会让ai去发起，需要代码先去跑一个字典拿到响应和测试结果等信息之后再让ai去写payload再继续测试。

于是这个Agent也是让代码去写的

3.Exploit Agent

这个是一个核心Agent，需要通过我们的提示词以及payloads字典扫描的结果去进一步构造payload去进行渗透，也是ReAct模式的主要应用的地方，需要拿到上一次对话构造的payload进行升级。

具体请况就是，跑的字典不可能直接进行一些渗透，比如sql注入跑字典的时候拿不到表明列名等信息就无法注入，ai可以根据我们拿到的信息一步步加强渗透。然后面对一些比如字典的payload被过滤了的请况，跑字典没有跑出结果，ai也可以尝试去进行绕过。

4.Analyzer Agent

这个Agent主要作用就是在Exploit Agent之前根据响应去判断是否存在漏洞，如果漏洞已经被证实了之后停止ReAct进行测试，如果跑完规定的ReAct还是没有最终的结果也是需要这个Agent去分析测试结果判断漏洞的。

本来跑完字典也需要这个Agent先判断一遍，但是还是坚持跑字典的时候不要直接命中漏洞，万一可以闭合sql语句但是过滤的特别死那不就小丑了吗。还是要拿到所有结果才能判断漏洞是否存在，以及根据测试结果才能进行漏洞评级。

5.Validator Agent

验证Agent，在AnalyzerAgent跑完之后我们会拿到目前测试出来的各个漏洞信息，但是不能作为漏洞存在的证据，需要我们使用代码再验证一遍，如果验证通过了就可以继承之前得到的漏洞评级等信息，没有过则后面再说

目前这个模块还不需要考虑

6.Reporter Agent

检测到漏洞如果不写个报告出来，作品赛估计是初赛就被刷了。使用ai结合前面分析到的信息按照固定模板写一个报告而已，属于最简单实现的功能。

6.LLM配置

我真的是服了，果然不会开发还是不行吗。为了装的我的平台可以支持多个LLM的api，结果花了好长时间就为了读取配置然后对特定的模型进行连接。作用和自己写几行代码接入LLM一样。原来项目接入ai不是像api的部署文档里面一样。

接入LLM主要使用的是两个文件分别是

/backend/app/core/config.py

/backend/app/agent/llm/service.py

其中config.py的作用只有一个，就是读取LLM的配置，包括过api_key base_url moduls等信息，然后支持从.env文件中读取和默认配置以及代码配置三种方式。

service.py的作用就是根据我们config.py读取到的配置然后返回对应的LLM模型。

只能说没有多大用好吧，代码量和功能不成正比。

7.SQL注入模式

我就说随着我的开发我会发现越来越多的SB的问题的，也算是拿时间交学费了。

我之前就觉得为什么要那么多个工具让ai去调用，直接主流程一套下来不就行了吗。一开始我将爬虫写到主程序里面，每次都会爬虫就省去了每次ai都去自己调用。

结果等我做到sql注入的时候才发现工具真正的用处，我发现ai不能做的事情很多，比如

  • 联合注入我不能拿整个响应界面给ai，Token遭不住
  • 盲注需要很多个payload，ai肯定不行，除非我使用沙箱让ai自己写代码，但是不够稳定，而且Token也是个问题

还有很多问题都不能完全使用ai生成的payload，然后sql注入用ai其实挺难的，至少比模板注入那些攻击方式比较局限的要难。我目前的解决方式就是，盲注写一个工具，联合注入写一个工具，总之就是ai生成的payload无法完成的全部都写个工具去跑脚本，ai的主要作用在于生成一些绕过waf的payload。

比如盲注在ai生成绕过waf的payload

然后之前预测的ReAct模式估计也需要改变一下，然后整个流程中ai调度的占比可能要大一点。
