2.漏洞分析

对于struts2框架的请求，我们可以看到其流程

自己太菜了，还是老老实实的把这个流程过一遍吧

1. 请求入口：Servlet Filters

  • ActionContextCleanUp
    负责清理当前线程的 ActionContext 信息，防止内存泄漏。在多过滤器环境下（如与 SiteMesh 整合）很重要。
  • FilterDispatcher（Struts 2.0~2.1）或 StrutsPrepareAndExecuteFilter（2.3+）
    这是 Struts2 的核心入口，所有请求都会经过它。

2. Struts 核心控制层

  • ActionMapper
    将 HTTP 请求映射到具体的 Action（判断是否需要执行某个 Action）。
  • Configuration Manager
    加载 struts.xml 及其他配置文件，管理 Struts2 的运行时配置。
  • ActionProxy
    创建 Action 的代理对象，协调拦截器栈与 Action 的执行。

至于ActionMapper往回传的那个是因为判断出来这个请求不应该由struts2进行处理

3. 拦截器栈（Interceptors）——漏洞高发区

这个就是我们代码审计的重点，我们看到拦截器分为两部分，中间由Action隔开，然后我们注意一下其实是进栈和出栈

1. 预处理（Before Action）

在 Action 方法执行之前工作，为 Action 的执行准备“上下文”和“数据”。

  • 参数处理：最经典的 ParametersInterceptor，它将 HTTP 请求中的参数自动提取并设置到 Action 的对应属性中。你不需要在 Action 里手动写 request.getParameter("name")。
  • 数据验证：ValidationInterceptor 根据你定义的规则（注解或XML）验证输入数据的合法性。
  • 工作流控制：WorkflowInterceptor 根据验证结果决定是继续执行 Action 还是跳转到“输入”页面。
  • 安全控制：自定义的 AuthenticationInterceptor 可以检查用户是否登录，是否有权限访问当前 Action。
  • 文件上传：FileUploadInterceptor 处理 multipart 表单，将上传的文件封装成方便操作的对象。

然后经过Action，Action主要是对我们这个请求进行处理，是主要的业务逻辑的代码

然后返回的结果回进行模板渲染

2. 后处理（After Action）

在 Action 方法执行之后、视图渲染之前/之后工作，主要用于清理和收尾。

  • 资源清理：关闭数据库连接、清理线程局部变量（防止内存泄漏）。
  • 日志记录：记录 Action 执行的最终结果和耗时。
  • 数据加工：对 Action 执行后产生的数据进行最后的修饰。

然后这是大佬的总结，感觉很生动

  1. HTTP请求经过一系列的过滤器，最后到达 FilterDispatcher 过滤器。
  2. FilterDispatcher 将请求转发 ActionMapper，判断该请求是否需要处理。
  3. 如果该请求需要处理，FilterDispatcher会创建一个 ActionProxy 来进行后续的处理。
  4. ActionProxy 拿着HTTP请求，询问 struts.xml 该调用哪一个 Action 进行处理。
  5. 当知道目标Action之后，实例化一个ActionInvocation来进行调用。
  6. 然后运行在Action之前的拦截器，图中就是拦截器1、2、3。
  7. 运行Action，生成一个Result。
  8. Result根据页面模板和标签库，生成要响应的内容。
  9. 根据响应逆序调用拦截器，然后生成最终的响应并返回给Web服务器。

看完整个流程以后，框架的主要漏洞点就在拦截器中

我们可以去struts-default.xml中找到拦截器栈
